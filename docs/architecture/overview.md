# Architecture Overview

## Hexagonal Architecture

semantic-release is built on the Hexagonal Architecture (Ports and Adapters) pattern. This decision drives the entire codebase structure.

### Layers

**Domain** (`internal/domain/`) — Pure value types and business rules with zero external dependencies. Includes Version, Commit, Project, ReleaseType, ReleasePlan, and Config types. All logic here is deterministic and trivially testable.

**Ports** (`internal/ports/`) — Go interfaces that define the contracts between the application and the outside world. Examples: GitRepository, CommitParser, TagService, ReleasePublisher. These interfaces are the architectural boundaries.

**Application** (`internal/app/`) — Use cases that orchestrate domain logic through ports. The ReleasePlanner builds a plan, the ReleaseExecutor carries it out. These services receive their dependencies via constructor injection.

**Adapters** (`internal/adapters/`) — Concrete implementations of ports. The git CLI adapter, GitHub REST adapter, Viper config adapter, and Cobra CLI adapter all live here. Adapters are freely replaceable without touching core logic.

**DI Container** (`internal/di/`) — A simple, explicit container that wires adapters to ports and creates application services. No reflection magic — just constructor calls organized in one place.

## Key Design Decisions

### Strategy Pattern for Project Discovery

Project discovery uses a strategy-based approach with three implementations:
1. `WorkspaceDiscoverer` — parses `go.work`
2. `ModuleDiscoverer` — scans for `go.mod` files
3. `ConfiguredDiscoverer` — reads from config

These are composed via `CompositeDiscoverer` which tries each strategy until one returns results.

### Template Pattern for Tags

Tag formatting uses Go templates, making the naming convention fully configurable. The same tag service handles both formatting (project + version → tag name) and parsing (tag name → project + version).

### Decorator Pattern for Dry Run

Dry-run behavior is implemented at the executor level rather than through decorators on individual operations. The executor checks `plan.DryRun` before performing any mutation, which keeps the code simple and avoids a proliferation of wrapper types.

### Impact Analysis

For monorepos, the `PathBasedImpactAnalyzer` maps each commit's changed files to projects by comparing file paths against project path prefixes. When dependency propagation is enabled, it additionally marks downstream projects for release when their dependencies change.

## Data Flow

```
CLI Command
    → DI Container (wires dependencies)
    → ProjectDetector.Detect() (discovers projects)
    → CommitAnalyzer.Analyze() (fetches + parses commits)
    → ReleasePlanner.Plan() (calculates versions per project)
    → ReleaseExecutor.Execute() (tags, publishes)
    → CLI Output (human or JSON)
```

## Testability

Every external concern (git, filesystem, GitHub API, time) is behind an interface. Unit tests use mocks generated by uber-go/mock. The domain layer requires no mocks at all — it's pure functions on value types.
